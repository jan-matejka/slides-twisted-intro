Twisted

Event-driven networking engine in Python licensed under MIT license.

Why should you care?

  • 10k problem
  • IO bound applications
  • async IO
  • without thread-like API
  • cooperative multitasking
  • elegant design of protocol implementation

History

  • Starts at year 2000
  • @glyph, the creator of Twisted, was working on a text-based multiplayer game called Twisted Reality
  • in java, with threads (which deadlocked, caused bugs and scaled poorly)
  • Alternative found in python select module
  • Worked much better and gradually evolved into the netwoking engine.

What is Twisted?

  • Framework for asynchronous programming
  • It's not just asynchronous layer in between socket and synchronous code
  • This asynchronny fundamentally affects how you write and think about the code
    • I/O performed via callbacks which makes the code nonlinear
    • troublesome use of libraries without async interface

Features

  • Reactor Loop
  • Deferred
  • Transport & protocol separation
  • Protocol implementations
  • Foreign Loop support

Reactor Loop

Deferred

  • An object returned instead of the result
  • Attached functions in callback/errback chains.
  • When the result actually arrives the callback chain gets executed
  • Hard to describe, easier to read the source code

Deferred callback chains

# Deferred example
def errback(reason):
    print "Oh noes: {0}".format(reason)
    sys.exit(1)

d = twisted.web.client.Agent().request('GET', url, Headers())
d.addCallback(lambda response: r.deliverBody(YourProtocol()))
d.addErrback(errback)
d.addBoth(lambda _: reactor.stop())

reactor.run()

Transport & protocol separation

  • Layer abstraction/encapsulation similar to TCP/IP model
  • example: json-rpc - http/netstrings
  • 3 main parts
    • Transports
    • Factories
    • Protocols

flow / object lifetime

  1. Transport (with a factory) listens on TCP socket
  2. When client connects, factory creates the Protocol object
  3. Transport delivers data into the Protocol
  4. Protocol handles the data
  5. When connection is terminated, protocol object is destroyed

Protocol implementations

  • Big advantage is many existing implementations of standard protocols
  • like HTTP, IRC, XMPP, SMTP, etc
  • using them is just a matter of implementating the prepared interfaces:
    1. deriving the protocol interface
    2. implementing the event methods
    3. attaching the protocol to transport
    4. and starting the reactor

IRC protocol example

class IRCClient(basic.LineReceiver):
    def userJoined(self, user, channel):
        """Called when I see another user joining a channel.
        """
        pass

    def topicUpdated(self, user, channel, newTopic):
        """In channel, user changed the topic to newTopic.

        Also called when first joining a channel.
        """
        pass

IRC protocol example

class YourIRCBot(IRCClient):
    def userJoined(self, user, channel):
      self.transport.write("Hello " + user)

  class BotFactory(t.i.p.ReconnectingClientFactory):
    protol = YourIRCBot

  reactor.connectTCP("irc.freenode.net", 6667, BotFactory())
  reactor.run()

Foreign Loop support

  • Twisted can integrate with foreign event loops, such as those of GTK+, Qt and Cocoa (through PyObjC)
  • This allows using Twisted as the networking support layer in graphical user interface (GUI)

Caveats

  • Database access
    • Kinda complicated due to non-blocking requirement
    • but there is evennia MUD project somehow using Django ORM
    • SQLalchemy can be used w/ deferToThread
    • twisted.enterprise.adbapi asynchronous DB-API v2.0 (raw SQL)
    • Twistar Active Record for Twisted
  • Twisted's unit test runner - Trial
    • some errors manifest in later test cases
    • incorrect tests may hang forever
  • Twisted as a platform
    • Most of the libraries are synchronous -> shortage of libs that play well with twisted
    • Relatively small community and slow progress on new standards
